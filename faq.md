# FAQ

## Q1: 为什么把 `concurrency` 从 10 提到 20，`wall_elapsed` 没有明显下降？

A: 说明当前不是“并发不够”，而是已经接近系统上限。并发继续加大，只会增加排队和竞争，不会线性缩短总耗时。

## Q2: 为什么把 `insert_batch_size` 从 8 提到 16，`wall_elapsed` 只小幅变化？

A: `batch` 变大后，单次 `client.insert()` 变重了，`operations` 变少了，但单次 `TP` 也变大了。最终只是减少了一部分 RPC 和调度开销，所以 `wall_elapsed` 只小幅下降。

## Q3: 为什么加了 HDF5 预取，`wall_elapsed` 变化还是不大？

A: 预取只能隐藏“读文件阻塞 worker”的问题。如果当前主瓶颈在 `rpc_*` 或 Python 对象构造，预取不会带来明显收益。

## Q4: 当前的 `wall_elapsed(s)` 到底包含什么？

A: 它表示并发执行阶段的墙钟耗时，包含 HDF5 读取、records 构造、Milvus 请求、线程等待和进度条更新；不包含数据生成、建库建表、建索引、`load_collection` 和最终 `flush`。

## Q5: 为什么 `rpc_TP*` 比 `prep_TP*` 更高？

A: 说明真正重的部分在 Milvus 写入链路，而不是单纯的本地 dict 构造。当前多向量 `struct-float32` 模式下，客户端构造成本也不低，但 RPC 仍然是更重的一段。

## Q6: 在我这组压测里，`concurrency` 从 10 提到 20、`insert_batch_size` 也从 8 提到 16，但 `wall_elapsed` 还是只小幅变化。结合这组现象，当前瓶颈更像在哪？

A: 更像是 Milvus 写入 RPC / 服务端写入链路已经接近饱和，或者至少进入了并发加不上去的区间。客户端本地构造有成本，但不是主瓶颈。

## Q7: 为什么不是“并发越大，wall 越小”？

A: 因为并发只能把任务同时发出去，不能凭空增加系统总处理能力。只有当客户端、网络、Milvus 服务端这条链路里还有空闲余量时，加并发才会更快。你这组结果里，`concurrency` 从 10 提到 20，`wall_elapsed` 没明显下降，说明系统不是“请求不够多”，而是“总处理能力已经接近上限”。再加并发，通常只是让更多请求在服务端侧排队等待处理，换不来更短的总耗时。

## Q8: 结合前面几组结果，`concurrency` 和 `insert_batch_size` 调整后都没有带来大幅收益，接下来更该优先调什么？

A: 优先试更大的 `insert_batch_size`，例如 `16`、`32`；并发建议停在 `8` 到 `12` 这一段做比较，不要继续盲目拉高。`prefetch_batches` 可以保留为 `concurrency * 2` 的默认值。

## Q9: `dict` 构造还能像换 `orjson` 那样大幅优化吗？

A: 不能类比。这里慢的不是 JSON 编码，而是必须创建大量 Python 对象，尤其是多向量里每条记录的 `tokens` 小 dict。只能做局部优化，难有数量级提升。

## Q10: 有没有一个简单的判断原则？

A: 有：
- `read_*` 高，且 `wall_elapsed` 对并发不敏感，偏向读文件瓶颈。
- `prep_*` 高，偏向 Python 对象构造瓶颈。
- `rpc_*` 高，且并发增加后吞吐不涨，偏向 Milvus 写入链路瓶颈。

## Q11: 我同时开了两个窗口，跑的是同一组参数，两个窗口的进度条速度几乎一样，这说明什么？

A: 这更像是“单个窗口对应的链路已经到上限”，不一定是 Milvus 全局打满。也就是说，问题可能卡在单个客户端进程、单条连接、单个 worker 组，或者服务端某个入口路径上，而不是整个集群都没有余量。要判断是不是全局瓶颈，要看两个窗口的**合计吞吐**：如果合计吞吐明显上升，说明服务端整体还有余量；如果合计吞吐基本不变，才更像是服务端总链路饱和。

## Q12: 为什么单窗口把 `concurrency` 调高，总吞吐不怎么变，但同时开两个相同参数的窗口，总吞吐却能上去很多？

A: 因为当前脚本用的是线程池，`batch_to_records()` 这类 Python 对象构造主要是 CPU 活，还会受 GIL 影响。线程多了，不代表这段代码能真正并行跑；它更像是在一个进程里轮流执行。相反，两个窗口通常是两个独立进程，每个进程各自有自己的 GIL 和 CPU 调度，所以双窗口合起来的总吞吐可以明显上升。这个现象说明，单窗口里卡住的不一定是 Milvus 全局服务端，更可能是客户端单进程的 Python 预处理上限。


## Q13: 我把插入脚本切到 `process` 模式后，`concurrency` 从 10 提到 20，为什么 `wall_elapsed` 还是只小幅变化？

A: 这说明客户端侧的 `read_*` 和 `prep_*` 已经很低了，`process` 也基本把 GIL 和 Python 构造成本压下去了；真正限制总耗时的是 `rpc_*`。换句话说，20 个进程并没有把服务端处理能力再抬高一截，只是让更多请求同时进入 Milvus 写入链路，结果更多时间花在服务端侧等待处理，所以 `wall_elapsed` 只能小幅下降，`rpc_TP*` 还会变高。

## Q14: `rpc_TP*` 变高但 `success_rows/sec` 只小涨，应该怎么理解？

A: 这表示并发继续加大后，吞吐已经接近上限，新增并发主要是在把队列拉长，不是在把总处理能力拉高。更直白地说，系统还能继续接收请求，但每个请求拿到服务端资源的等待时间变长了，所以单次 RPC 更慢，整体吞吐只多一点点。
